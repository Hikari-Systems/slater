Thinking about compression techniques for data structure encoding beyond adjacency blocks

I can't actually reach that repo — github.com/Hikari-Systems/slater isn't visible to me (private, or at least not indexed/fetchable), and my fetch tool refuses URLs that haven't come back from a search. So take the below as a catalogue rather than a code review; if you paste format.md/the layout module or the encoder for the adjacency block, I'll map it onto the actual shapes you've got.

That said, the question generalises well, because what you've bought with EF+RLE is a specific property, and it's worth naming it precisely before shopping for more of it:

access/rank/select in the encoded domain, with O(1)-ish random access and no materialised intermediate. EF gives you select(i) (the i-th neighbour) and rank(x) (how many neighbours < x, i.e. a membership/intersection primitive) straight off the high/low bitplanes. RLE gives you the same for runs. Everything below is chosen because it has that same shape, not merely because it's small.

Where the equivalents live

Term/string dictionary (IRIs, literals, labels) — usually the fattest part after adjacency, and the one most people leave as "decompress a page then memcmp".

  • FSST (Boncz/Neumann/Leis, VLDB'20) is the direct analogue: a 255-entry symbol table, strings compress individually, decompression of a single string is a tight loop with no block context. You can even do prefix/equality predicates on the compressed bytes if you compress the probe with the same table. Typically 2–3× on IRIs, ~5× on JSON-ish literals, and a few ns/string.
  • Rear-coded / front-coded dictionary with a bucket header: sort terms, store every 16th (or 32nd) verbatim, the rest as (shared-prefix-len, suffix). id→term is O(bucket) not O(n), term→id is a binary search over headers plus a linear scan. It's the string-world cousin of "sampled EF high bits". Pairs beautifully with FSST on the suffixes.
  • If you want term→id without storing the keys at all: a monotone minimal perfect hash (PTHash/RecSplit) at ~3 bits/key, plus a verification byte or an id→term round-trip to reject non-members. Brutal win when the dictionary dominates and you control membership elsewhere.
  • If you need prefix/range queries (SPARQL STRSTARTS, autocomplete, IRI-namespace scans): a succinct trie — LOUDS or SuRF-style — gives you traversal in the encoded form. This is the "no decompress" answer for prefix predicates.

Sorted-set / posting-list-ish structures (indexes, type sets, deletion masks, id remaps)

  • Roaring is already exactly the property you want: intersect/union/rank/select on the encoded form, and it's structurally the same idea as your block RLE (dense bitmap vs sorted-array vs run container, chosen per 2¹⁶ chunk). If your adjacency RLE already partitions per block, you're one hop from a Roaring-shaped container discriminator, and you get cheap boolean algebra between adjacency and filters for free.
  • RRR bitvectors where you need rank/select on a compressible bitvector and don't want the 25% overhead of a plain rank index — entropy-compressed but O(1) rank.

Per-edge payloads: edge labels, weights, timestamps, provenance

  • Keep them in parallel arrays aligned to the same block boundaries as the adjacency, bit-packed with per-block frame-of-reference. Bit-packing is already the ideal "no decompress needed" codec: element i in a block is (word >> shift) & mask. No block scan, no state.
  • Monotone-per-adjacency-list timestamps → EF again. Non-monotone → delta + FOR + bit-pack.
  • Floats (weights, scores) → ALP if you want random access; Gorilla/Chimp are streaming-oriented and force a scan, so they violate your constraint.

Typed adjacency: "neighbours of v with edge label L"
This is the one where there's a genuinely elegant answer people usually miss. A wavelet tree over the edge-label sequence, aligned with the adjacency array, gives you access(i), rank_L(i), select_L(k) in O(log σ) on the encoded form. That means "the k-th L-labelled neighbour of v" is a couple of rank/select calls, and label-filtered degree is a rank difference — no scan of the list, no decode. It's the same trick DBMS people use for wavelet-tree-based tries; in a triple store it's the difference between SPO and SPOL indexes existing at all. If you currently store one adjacency block per (vertex, label) pair, a wavelet tree lets you collapse those into one list and still answer label-filtered queries in the encoded form — often a big win on graphs with many sparse labels.

Offsets / CSR row pointers — monotone by construction, so EF (or partitioned EF with per-partition parameters, PEF-style) directly. If you've EF'd the neighbours but left the offsets as a plain Uint32Array/BigUint64Array, that's usually a free 60–70% off the offsets with no loss of select.

Cardinality/statistics for the planner — sketches are the "compressed and directly usable" idea taken to its logical end: HLL for distinct counts, CM/Count sketches for skew. You never decompress those; you query them.

The bit-packing point, concretely

Since bit-packing underlies most of the above and is the cheapest thing to add, the whole primitive is this — random access, no block decode, no state:

ts
type PackedBlock = Readonly<{
  words: BigUint64Array;   // or two Uint32Array lanes if you'd rather stay in 32-bit land
  width: number;           // bits per value, 1..64
  base: bigint;            // frame-of-reference minimum for this block
}>;

const readPacked = (b: PackedBlock, i: number): bigint => {
  const bitPos = i * b.width;
  const word = bitPos >>> 6;
  const shift = BigInt(bitPos & 63);
  const mask = (1n << BigInt(b.width)) - 1n;

  const lo = (b.words[word] >> shift) & mask;
  const spill = (bitPos & 63) + b.width - 64;
  const hi =
    spill > 0
      ? (b.words[word + 1] << (BigInt(b.width) - BigInt(spill))) & mask
      : 0n;

  return b.base + (lo | hi);
};

Everything downstream (edge weights, timestamps, dictionary suffix lengths, wavelet-tree levels) is a specialisation of that plus a rank/select index. Note the spill branch: if you pick width per block from {1,2,4,8,16,32} you eliminate it entirely at a few percent space cost — often the right trade, because the branch is unpredictable and this sits on your hot path.

The awkward bit: it's the writeable branch

The uncomfortable truth about all of the above — EF included — is that none of it is updatable in place. The standard shape, and I'd bet it's already implicit in what you've built:

  • immutable succinct segments,
  • a small mutable delta layer per segment (plain sorted arrays / a hash map, uncompressed, bounded size),
  • reads = merge-on-read across (segment, delta),
  • compaction rebuilds the segment when the delta exceeds some fraction of it.

Two refinements that matter specifically for succinct structures:

  1. Deletions as a Roaring tombstone bitmap over positions, rather than rewriting the EF. Rank on the tombstone bitmap gives you the live rank, so select composes: liveSelect(k) = select(k + rank_deleted(...)). You keep the "no decompress" property through deletes.
  2. Slack in the EF low bits. If you size the low-bit width for the expected universe rather than the current max, you can absorb appends into a list without re-encoding — a "relaxed EF" with a rebuild threshold. Cheap, and it turns most appends into O(1).

If you tell me which of these components exist in slater today and roughly what their on-disk shapes are, I can be much more specific about which one to reach for first — my prior, based on where the bytes usually go in a store that's already solved adjacency, is that the term dictionary is now your biggest remaining lump, and FSST + rear-coding is the highest ratio of win to disruption. The wavelet tree is the more interesting idea but it changes your query planner, not just your codec.

The diagnosis, precisely

grep -rn "BlockCodec::Raw" across the workspace returns exactly one hit: nodedegree.rs:63. Everything else — topology.csr.blk, node_props.blk, edge_props.blk, node_labels.blk, reltype_*.post, isam, prop_hist.blk — is zstd blocks in the BlockCache, i.e. decompress-to-use, and resident in decompressed form. codec.rs's doc comment ("All .blk files are split into fixed-size raw blocks, each compressed independently with zstd") is now literally false, and that falsity is the opportunity.

More importantly, the property you actually won in v4 wasn't "smaller". It was:

  • decompress-free fault (Raw container ⇒ pread + parse),
  • compact resident form (never a dense u32 array),
  • O(1) random access in the encoded form (select₁ + packed low bits),
  • per-chunk codec chosen by measurement, with zstd_margin penalising the codec that costs a decompress.

That last bullet is the crown jewel and it's currently locked inside degree_ef.rs. Generalise it — enum ChunkKind { Constant, Ef, Rle, RawU32, ZstdDense } plus "pick the smallest, but tax the one that costs a decompress" is a generic plane codec, and almost every hot structure in slater is a plane in disguise.

Ranked candidates

1. topology.csr.blk — by a mile the biggest, and the one with real query consequences

This is the one that made me sit up. Current forward record:

uvarint(count) ‖ u8(flag) ‖ uvarint(edge_id_base) ‖ count × ( uvarint(reltype) ‖ uvarint(neighbour) )

Three problems, all structural:

  • neighbour is an absolute uvarint — on a 91.6M-node graph nearly every id is ≥ 2²¹, so that's 4 bytes each, and there's no delta encoding at all. You're leaning entirely on zstd to notice.
  • The planes are interleaved (AoS), so the reltype varints sit between consecutive neighbour varints. That destroys any chance of a monotone codec over the neighbours and forces you to decode reltypes you don't want.
  • It is sequentially-decodable only. decode_adj_into is a varint walk. There is no k-th-neighbour, no successor, no rank.

And the third one bites hard. exec.rs:485:

rust
topology::decode_adj_into(&rec, outgoing, |a| {
    if !core_removed.contains(&a.edge.0) && delta_keep(&a) { push(a)?; }
    Ok(())
})?;

A typed expand (v)-[:T]->(?) on a hub with 5M outgoing edges decodes all 5M varint pairs and then filters. That's the single hottest loop in the engine and it is doing full decode-then-discard.

The fix is exactly the degree_ef move, one level up. Note that your builder already sorts forward edges by (final_src, final_dst, prov_edge_id) (build_external.rs:1783) — so the neighbour list within a record is already non-decreasing. It is already an EF-shaped sequence and you're storing it as absolute varints. Likewise reverse: edge ids ascend and are assigned src-major, so reverse neighbours are ascending too.

Split the record into planes and code each with the plane codec:

planeshapecodec
neighbournon-decreasing over [0, N)Elias–Fano (handles non-decreasing natively — high[i]+i stays strictly increasing even with duplicate parallel edges, so no fudge needed). PEF if you want to exploit Wikidata's heavy clustering.
reltypetiny alphabet, long runsRLE, or bit-pack at ⌈log₂ σ⌉, or a wavelet tree if you want rank_T
edge_idforward: base + k (already free, v5). reverse: strictly ascending sparse subsetEF

What that buys, in the encoded domain, with no decompress:

  • neighbourAt(v, k) → one select₁.
  • hasEdge(v, u) → EF successor + equality. O(1)-ish instead of O(deg(v)) — this is what makes triangle counting, common-neighbour and cyclic-pattern matching viable.
  • List intersection by leapfrog/SvS using successor to skip — the classic quasi-succinct posting-list join. Two hub lists intersect in O(min) skips rather than O(sum) decodes.
  • adj_count_by_reltype(v, T) for the planner, free.

And there's an elegant composition available on the typed-expand problem. Change the builder's forward sort key from (final_src, final_dst, prov_edge_id) to (final_src, reltype, final_dst):

  • edge ids are still assigned in forward-CSR order, so v5's edge_id = base + k derivation survives untouched;
  • each record now has reltype-major runs, so a 3-entry per-record directory (reltype → [lo, hi)) lets (v)-[:T]->(?) seek straight to the slice;
  • each run's neighbours are ascending, so EF per run;
  • reverse records sorted (final_dst, reltype, final_src) get the same directory, with per-run-ascending edge ids → EF.

That turns your worst case (typed expand from a hub) from "decode 5M varints" into "one directory probe + EF-select over 12 elements". This is the biggest single lever in the codebase.

Honest caveat: raw EF may well be larger on disk than zstd-over-varints for the low-degree bulk of the graph — average degree here is only ~16, so d·(2 + log₂(N/d)) ≈ 24.4 bits/edge vs a 4-byte varint that zstd then squeezes. Which is precisely why you don't guess: reuse DegreeCodecOpts::zstd_margin and let the per-record selector decide, with ZstdDense taxed for the decompress it costs. You already built the apparatus for exactly this argument.

2. The resident sorted u64 key columns — the writeable-branch-specific win

l0_offheap.rs and segment.rs both say the same thing: "binary-searching a resident sorted u64 key column", "Resident cost is the key columns (8 B/entry)".

That is a monotone u64 sequence whose only operation is predecessor/successor search. It is the textbook EF case, and EF's successor is the binary search you're already doing. For a segment holding ~1M keys over a 91.6M id universe: 2 + log₂(2²⁷/2²⁰) ≈ 9–10 bits/entry against 64. A ~6–7× cut in resident memory, matching the degree win almost exactly.

This matters disproportionately on the writeable branch because resident key-column bytes scale with segment count, and segment count is the thing that grows between compactions. It's also the cheapest to implement — you already have EfChunk; the key column is a single EF sequence with no record framing. If I were sequencing this work, this is what I'd do first: smallest diff, clearest win, directly relieves the pressure that the segmented-core design creates.

3. reltype_src.post / reltype_tgt.post — the canonical EF case, currently missed

From postings.rs: "ascending distinct source node ids… uvarint(count) ‖ uvarint(first) ‖ uvarint(delta)… — ascending ids ⇒ small deltas ⇒ the per-block zstd packs them tightly."

Sorted distinct id lists, used as driving sets that then get probed and intersected. This is literally what quasi-succinct indices were invented for (Vigna's paper is about exactly this). Delta+varint+zstd means you must decompress the block and walk the whole list to skip anywhere. EF (or Roaring, if you want cheap boolean algebra between a reltype set and a label set) gives you successor-based skipping and set intersection with zero decompress.

The "~8% of nodes that actually have a T edge" line in your docs is the tell: these are sparse sets over a dense universe, being used as filters. That's Roaring's home ground.

4. node_labels.blk — make label predicates a bit test

uvarint(count) ‖ count × uvarint(label_id), per node, in a zstd block. Every scan evaluates n:Label. DECISIONS.md:1375 already notes the reltype/label alphabets are small.

If the label alphabet is ≤ 64, the whole record is a u64 bitmask: labels(n) is one aligned load, n:Label is mask >> id & 1, and the file is a dense u64 plane in a Raw container — random-access by dense node id with no framing and no decode at all. Above 64, bit-pack a fixed-width plane with an overflow escape. Cheap, and it deletes a varint scan from the innermost predicate in the engine.

(The inverted labels.post you've got pencilled in for a later milestone: Roaring, obviously, not delta-varint.)

5. isam.rs — the LevelDB restart-point trick

Currently: block raw = uvarint(count) ‖ count × ( value(key) ‖ uvarint(id) ) (then zstd). So a point lookup binary-searches the sparse top level, then decompresses an entire block and linearly scans it.

Store the block Raw with prefix-compressed keys plus restart points every 16 entries (a small trailing offset array). Now you binary-search inside the block over the restart offsets and linear-scan only a 16-entry group — zero decompress on a point lookup. Add FSST for the key suffixes if the strings are fat (they will be, on node_Entity_wikidata_id) and you keep the block Raw while still getting 2–3× on the bytes. Front-code the resident top level while you're at it.

Given DECISIONS.md:1284 flags node_Entity_wikidata_id as near-unique over 91.6M nodes and BUILD-PERF-PLAN.md:373 mentions a 5.78 GB artefact in that neighbourhood, this one is worth real money.

6. node_props.blk / edge_props.blk — FSST, keeping your row-major design

Your row-per-entity choice is well-argued in the header and I wouldn't overturn it for a column store; the dominant read really is "materialise this entity for a RETURN projection". But you can have the no-decompress property without changing the layout:

FSST (Boncz/Neumann/Leis, VLDB'20) compresses each string against a 255-entry symbol table such that a single string decompresses in isolation, with no block context. Put the symbol table in the MANIFEST, store the property blocks Raw, and a value materialises with a tight symbol expansion instead of a zstd block decode. Typically 2–3× on IRIs, ~5× on JSON-ish literals, a few ns per string. Equality predicates can even be evaluated on the compressed bytes by compressing the probe with the same table.

The primitive, in the form you asked for

The two operations that unlock everything above are select (k-th) and successor (first ≥ x). Sketched over a plane rather than a class:

ts
type EfPlane = Readonly<{
  high: BigUint64Array;   // unary bucket bitmap: element i's 1-bit at high[i] + i
  low: BigUint64Array;    // packed ℓ-bit low parts
  lowWidth: number;       // ℓ = floor(log2(u / m))
  count: number;          // m
  selectSample: Uint32Array; // sampled select₁ index, every 64th one-bit
}>;

const readLow = (p: EfPlane, i: number): bigint => {
  if (p.lowWidth === 0) return 0n;
  const bit = i * p.lowWidth;
  const w = bit >>> 6;
  const s = BigInt(bit & 63);
  const mask = (1n << BigInt(p.lowWidth)) - 1n;
  const spill = (bit & 63) + p.lowWidth - 64;
  const lo = (p.low[w] >> s) & mask;
  const hi = spill > 0
    ? (p.low[w + 1] << (BigInt(p.lowWidth) - BigInt(spill))) & mask
    : 0n;
  return lo | hi;
};

/** k-th element. O(1): one sampled select₁ + one packed low read. */
const selectAt = (p: EfPlane, k: number): bigint => {
  const pos = selectOne(p.high, p.selectSample, k);      // index of the k-th set bit
  const highPart = BigInt(pos - k);                      // subtract the i offset
  return (highPart << BigInt(p.lowWidth)) | readLow(p, k);
};

/** First index whose value is >= x. The primitive behind hasEdge and leapfrog joins. */
const successorIndex = (p: EfPlane, x: bigint): number => {
  const hx = Number(x >> BigInt(p.lowWidth));
  // Skip straight to bucket hx: everything before it is provably < x.
  let i = hx === 0 ? 0 : rankOne(p.high, p.selectSample, hx) /* zeros before bucket hx */;
  while (i < p.count && selectAt(p, i) < x) i += 1;      // bounded by the bucket's occupancy
  return i;
};

const hasNeighbour = (p: EfPlane, u: bigint): boolean => {
  const i = successorIndex(p, u);
  return i < p.count && selectAt(p, i) === u;
};

The point of successorIndex is that it never touches elements outside the target bucket — that's what turns adjacency intersection from O(deg₁ + deg₂) decode into O(min) skips, and it's the operation that decode_adj_into structurally cannot offer.

Where I'd start

  1. EF the resident u64 key columns in l0_offheap.rs / segment.rs. Smallest diff, ~6× on the resident term that grows with segment count, no format-version churn on the core.
  2. Extract the plane codec out of degree_ef.rs into a generic plane.rs (Constant | Ef | Rle | BitPacked | Raw | ZstdDense + zstd_margin). It's already 90% written; it just doesn't know it's general yet.
  3. Re-plane topology.csr.blk (v6) with the (src, reltype, dst) sort-key change and the per-record reltype directory. This is the big one and it's where the query planner gains new powers, not just fewer bytes.
  4. reltype_*.post → EF/Roaring; node_labels.blk → bitmask. Both nearly free once (2) exists.
  5. isam restart points, then FSST on strings.

Happy to go deeper on any of these — the topology re-planing in particular has a fiddly interaction with slater-delta's adjacency fragments and the core_removed tombstone check in exec.rs that's worth thinking through properly before you commit to a v6.
